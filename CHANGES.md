# Changelog

## Unreleased

Switch the Matomo all-time window from `period=week, date=last260` to `period=year, date=last5`. Five pre-archived yearly buckets cost Matomo ~50× less memory than 260 weekly buckets, removing the HTTP 500 `Allowed memory size exhausted` failure mode that surfaced on high-traffic sites — `5 URLs × 260 weeks = 1,300 archive lookups` per bulk request was exhausting Matomo's 256 MB PHP limit. Yearly is also more accurate: weekly's 260-bucket sum loses ~13% to year-boundary rollup drift, while yearly matches Matomo's UI for "Last 5 years" exactly. First sync after upgrade will correct existing `mai_views_web` values upward by ~10–15% for high-traffic URLs. Raise default chunk size from 5 → 10 to match the original docblock intent now that the memory pressure is gone (`10 URLs × 5 years = 50 archive lookups` per request, still 26× under the configuration that crashed). Expose `matomo_bulk_chunk` as a Settings UI field next to the other Matomo credentials so operators can tune without writing a mu-plugin; the `mai_analytics_matomo_bulk_chunk` filter remains as the override path for backward compatibility. Provider error transient (`mai_analytics_provider_error`) now stores `{message, time}` JSON instead of a bare string, with a backward-compatible decoder in `Sync::get_last_error()` so legacy plain-string transients still resolve during the upgrade window. Surface provider errors as a dismissible admin notice on the Mai Analytics dashboard with a relative-age label (e.g. "Provider error: ... — 12 minutes ago") so operators can tell at a glance whether they're looking at a fresh failure or stale residue. Floor the lifetime total at the current trending value when writing `mai_views`, so the math invariant `total >= trending` holds across syncs even when the all-time fetch is briefly stale or failing — applies to both `ProviderSync` and self-hosted `Sync`.

## 1.1.3

Fix external-provider sync being a silent no-op since 1.1.0. `ProviderSync::sync()` overwrote `mai_analytics_provider_last_sync` with the current time at the top of the run and then re-read that same option a few lines later as the buffer-boundary cutoff, so `Database::get_distinct_objects_since()` was always querying for `viewed_at > now` and finding zero rows — meaning `mai_views_web` and `mai_trending` never updated for any object on Matomo/GA/Jetpack-backed sites, regardless of how many beacons fired or how often the cron ran. Capture the previous timestamp into a local before overwriting, mirroring the pattern in self-hosted `Sync::sync()`. Also fix `Cron::ensure_healthy()` and `Cron::maybe_fallback_sync()` reading `mai_analytics_synced` unconditionally — that option is only written by self-hosted `Sync`, so external-provider sites had a permanently-stale staleness check that hammered the fallback sync on every beacon. Both helpers now branch on `data_source` and read the option each sync path actually updates, matching `Health.php`. Fix `wp mai-analytics sync` reporting `Last sync: never` in external-provider mode for the same reason — the CLI's success line read `mai_analytics_synced` instead of branching on data source, making the post-sync timestamp meaningless for Matomo/GA/Jetpack sites. Also close the sync boundary race window in both `ProviderSync::sync()` and self-hosted `Sync::sync()`: both previously wrote `time()` to the last-sync option at the START of the run AND a fresh `time()` at the END, so buffer rows inserted during the run landed with `viewed_at` between sync_start and sync_end, fell behind the next sync's boundary (= sync_end), and were stranded in the buffer permanently. Capture `$started_at` once at the top and write that same value at finish, so the next sync's boundary is the previous run's start time and rows inserted during the run get processed.

## 1.1.2

Populate the dashboard's Posts-tab Author filter from authors of viewed posts, not from `mai_views` in `usermeta`. The previous source was empty on most sites because author archives are rarely linked, often disabled by SEO plugins, and excluded from tracking when the visitor has `edit_posts`, leaving the dropdown empty even when the site had thousands of viewed posts with attributed authors. Convert the Author filter to a Tom Select multi with ajax search via `/admin/search?type=author` (also fixed to query authors-of-viewed-posts), so big multi-author sites can pick any combination of authors without scrolling a 100+ option dropdown. Gate the Post Type and Taxonomy filter dropdowns on "has views > 0" too, so dashboards never offer filters that would return empty tables, and cache the whole `get_filters` payload in a 5-minute transient since filter membership changes far slower than the underlying view counts. Add `?subtab=posts|terms|authors|archives` so the Mai Analytics dashboard sub-tabs are deep-linkable; the active sub-tab now ships with a real `href` and is server-rendered with the correct active class, and the JS updates the URL via `replaceState` on click. Scope the active-tab class swap to `.mai-analytics-tabs` so clicking a sub-tab no longer strips the active state from the top-level Dashboard tab. Drop the dashboard's top-10 horizontal bar chart and its Chart.js dependency — it duplicated the precise, sortable table directly below it without surfacing anything new; removing it cuts a CDN script and ~100 lines of code. Drop the "Filtered by:" tag row underneath the filters; it duplicated the dropdowns themselves and toggling its `display:none` on every filter change caused visible CLS as the table jumped up and down. Convert every filter dropdown to Tom Select for visual uniformity (post type, taxonomy, term, author, publish dates) and rework the filter row as a `repeat(auto-fit, minmax(180px, 1fr))` CSS grid so cells share width, fill row height, and wrap naturally — no more skinny number input or oversized tag search. Disable the search input on static Tom Selects (`controlInput: null`) since their option lists are bounded short; click + arrow-keys still work, and the placeholder is restored via a `data-placeholder` ::before so the empty state still reads "All Post Types" / "All Taxonomies" / "All Publish Dates." Hide the chevron when an item is selected so the clear-button × no longer overlaps it. Nest the Custom Days input inside the Publish Dates grid cell so it shares space instead of consuming a full column. Extract the Publish Dates filter into a `PublishedDaysFilter` class that owns its Tom Select, debounced custom-days input, `is-custom` class toggle, and resolved day count, with a `getValue()` / `clear()` API — replacing two top-level functions, an inline listener, and a module-level integer. Switch the dashboard filter markup to BEM (`.mai-analytics-filters__field`, `--posts` / `--terms` modifiers, `__custom-days`, `__published`) so visibility logic reads the active-tab modifier directly instead of bespoke `.mai-analytics-filter-posts` classes.

## 1.1.1

Validate that Site Kit's `googlesitekit_owner_id` resolves to a real user before `wp_set_current_user()` so the GA4 REST `permission_callback` can't silently return `rest_forbidden` against a phantom user. Stale-owner sites now surface "Site Kit owner user (ID N) does not exist" with remediation guidance. Fix `SiteKit::get_unavailable_reason()` falling through to the "GA4 not connected" string even when the provider was available — it now matches its docblock and returns an empty string in the available case. Add `--chunk=N` to `wp mai-analytics warm` so one-off CLI runs can scale the Matomo bulk chunk without registering a `mai_analytics_matomo_bulk_chunk` filter in a mu-plugin. Distinguish `iterated` from `updated` in warm progress so the admin Warm Stats button and the CLI's verbose output show "N updated of M" when a batch's provider call fails and meta is preserved (the old "M updated" misled admins into thinking failed batches had succeeded). Replace the in-tree `error_log()` calls in all three providers with the shared `maithemewp/mai-logger` Composer package, gaining Ray and WP-CLI routing for free; the new `mai_analytics_logger()` helper is the public entry point. Decode HTML entities when capturing Matomo HTTP error response snippets so transient and debug-log output read as `›` and `«` instead of raw `&rsaquo;` / `&laquo;`. Refine the Warm Stats button hint copy so "leave this window open" reads as a parenthetical aside, not a peer status item.

## 1.1.0

Fix Matomo provider returning empty data so `mai_views` and `mai_trending` write real counts instead of `0` on Matomo-backed sites. The provider now expands paths to full URLs via `home_url()` (with `rawurldecode()` for Unicode-dash slug safety) and uses `period=day` / `period=week` against pre-built archives instead of `period=range`, mirroring the proven strategy from pre-bundle Mai Publisher's views class. Adds a `mai_analytics_views_years` filter (default `5`) for the all-time window and migrates the matching value from the `mai_publisher` option. Refactors `WebViewProvider::get_views()` to take an array of named windows and return per-window per-path counts, collapsing ProviderSync's two calls per batch into one bulk request (Matomo `API.getBulkRequest`, SiteKit per-window loop with single user-switch, Jetpack window loop over the cached per-post dataset). The Matomo provider chunks its bulk requests by path count to stay under `API_bulk_request_limit` and per-request memory budgets; chunk size is filterable via `mai_analytics_matomo_bulk_chunk` (default `5`). The admin Warm Stats button is now cursor-based and processes one batch per request via `ProviderSync::warm_batch()`, so large-site warms no longer hit Cloudflare's 524 timeout; it polls the new endpoint, shows per-batch progress, registers a `beforeunload` warning while running, and processes most-recent posts and terms first. The Matomo provider also enforces the math invariant `all_time >= trending` per path before returning, working around the current-incomplete-week gap in Matomo's weekly archives. Matomo failures now populate the `mai_analytics_provider_error` transient and surface response-body snippets in error messages, matching SiteKit and Jetpack's existing error-surfacing contract. See #5.
